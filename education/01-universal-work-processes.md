## 1. 모든 직군에 공통되는 보편적 업무 프로세스

### 1.1 프로젝트 생명주기 (Project Lifecycle)

모든 프로젝트는 직군에 관계없이 5단계를 거칩니다.

#### 1단계: 개시 (Initiation)
- **목적**: 프로젝트의 필요성, 범위, 목표 정의
- **주요 활동**:
  - 요구사항 수집
  - SMART 목표 설정
  - 이해관계자 식별
  - 초기 위험 분석
  - 팀 구성
- **산출물**: 프로젝트 헌장, 이해관계자 목록

#### 2단계: 계획 (Planning)
- **목적**: 실행 방법 상세 계획
- **주요 활동**:
  - 작업 분해 구조(WBS) 생성
  - 일정 수립
  - 자원 계획
  - 리스크 계획
  - 커뮤니케이션 계획
- **산출물**: 프로젝트 계획서, 마일스톤 정의

#### 3단계: 실행 (Execution)
- **목적**: 계획된 활동 수행
- **주요 활동**:
  - 작업 수행
  - 자원 배치
  - 일일 진행 회의
  - 품질 관리
  - 팀 협력
- **핵심**: 계획 대비 진행 상황 추적

#### 4단계: 모니터링 (Monitoring)
- **목적**: 진행 상황 추적 및 조정
- **주요 활동**:
  - 진행도 추적
  - 일정/비용 변동성 관리
  - 품질 점검
  - 리스크 평가
  - 변경 요청 관리
- **핵심 메트릭**:
  - 진행도 = (완료 작업 / 전체 작업) × 100%
  - 일정 건전성, 예산 건전성

#### 5단계: 종료 (Closure)
- **목적**: 공식 완료 및 학습 기록
- **주요 활동**:
  - 최종 검수
  - 인수인계
  - 문서화
  - 회고 회의 (Lessons Learned)
  - 팀 인정/보상

### 1.2 커뮤니케이션 워크플로우

#### 보고 체계

| 보고 유형 | 주기 | 내용 | 대상 |
|----------|------|------|------|
| **일일 보고** | 매일 | 어제 한 일, 오늘 할 일, 블로커 | 팀 |
| **주간 보고** | 매주 | 주간 목표 달성도, 주요 성과, 다음 주 계획 | 팀 리더 |
| **월간 보고** | 매월 | KPI 검토, 목표 달성도, 교훈 | 경영진 |

#### 효과적인 회의 원칙
1. **사전 준비**: 안건 사전 공지 (최소 24시간 전)
2. **진행**: 시간 엄수, 의제 순서대로
3. **기록**: 핵심 결정사항, 행동 항목 (누가, 언제까지)
4. **사후 관리**: 회의록 24시간 내 배포

#### 문서화 필수 항목
- 프로젝트 계획서, 요구사항 정의서
- 절차서, 매뉴얼, FAQ
- 회의록, 의사결정 기록
- 배운 점 (Lessons Learned)

---

### 1.3 의사결정 프로세스 (7단계)

1. **문제 정의**: 명확한 질문 형성
2. **정보 수집**: 데이터, 전문가 의견, 과거 사례
3. **대안 생성**: 최소 3가지 옵션 도출
4. **대안 평가**: 비용, 일정, 품질, 위험, 이해관계자 영향
5. **의사결정**: 결정 권한 확인 및 최종 결정
6. **전달 및 설명**: 누가, 언제까지, 무엇을, 왜
7. **모니터링 및 조정**: 결정 실행 추적

### 1.4 협업 패턴

#### 협업 모델

| 모델 | 설명 | 활용 |
|------|------|------|
| **순차적** | 한 사람 완료 후 다음 사람 | 단순 작업 |
| **병렬** | 여러 팀원 동시 작업 | 대규모 프로젝트 |
| **페어** | 2명이 함께 1개 작업 | 고위험 작업, 교육 |
| **검토 기반** | 작업 후 검토 | 품질 중심 |

#### 원격 협업 핵심
- 명확한 도구 활용 (Slack, Teams, Jira)
- 시간대 관리 및 비동기 커뮤니케이션
- 결과 기반 평가
- 철저한 문서화

---

### 1.5 작업 관리 기본 원칙

#### 작업 분류

| 단위 | 완료 기간 | 특징 |
|------|----------|------|
| **Epic** | 몇 주~몇 달 | 큰 목표, 여러 Story 포함 |
| **Story** | 며칠~1주 | 사용자 가치 제공 |
| **Task** | 반나절~며칠 | 기술/운영적 독립 작업 |
| **Sub-Task** | 몇 시간~1일 | 작은 단계 |

#### 작업 수명주기
```
To Do → In Progress → In Review → Done
```

#### 좋은 작업 할당의 조건
- **명확성**: 무엇을, 언제까지, 완료 기준
- **적정성**: 담당자 역량, 과부하 방지
- **책임성**: 담당자 수용, 진행 보고
- **추적성**: 정기 확인, 위험 신호 감지

---

### 1.6 품질 보증 (Quality Assurance)

#### 품질 관리 프로세스

1. **품질 기준 정의**: 수용 기준, 테스트 범위
2. **품질 보증 (QA)**: 프로세스 감시, 기준 준수 확인
3. **품질 제어 (QC)**: 검사 및 테스트, 결함 기록
4. **개선**: 결함 분석, 근본 원인 파악, 프로세스 개선

#### 테스트 유형

| 테스트 | 설명 | 시기 | 담당자 |
|--------|------|------|--------|
| **단위 테스트** | 개별 부품 검사 | 개발 중 | 개발자 |
| **통합 테스트** | 부품 간 호환성 | 개발 완료 | 개발팀 |
| **시스템 테스트** | 전체 시스템 | 통합 후 | QA팀 |
| **사용자 테스트** | 실제 환경 검증 | 배포 전 | 사용자 |

---

### 1.7 피드백 루프

#### 피드백 제공 원칙
- **구체성**: 구체적인 행동 지적
- **관찰 기반**: 추측이 아닌 사실
- **타이밍**: 신선할 때 제공
- **균형**: 칭찬과 개선점 균형
- **행동 지향**: 앞으로 어떻게 할지

#### 피드백 채널

| 채널 | 목적 | 주기 |
|------|------|------|
| **1:1 미팅** | 개인 성장, 관계 구축 | 2주/월 1회 |
| **팀 회고** | 팀 학습, 개선 | 스프린트/프로젝트 후 |
| **360도 피드백** | 다양한 관점 평가 | 연 1회 |
| **고객 피드백** | 사용자 요구 파악 | 지속적 |

## 2. 소프트웨어 개발자 업무 프로세스

### 2.1 개발 방법론

#### Agile/Scrum
- **스프린트**: 2주 단위 작업 사이클
- **Daily Standup**: 매일 15분 회의
- **Sprint Review/Retrospective**: 스프린트 종료 시

#### Kanban
- 연속 흐름 기반
- WIP(Work In Progress) 제한
- 시각화된 워크플로우

### 2.2 코드 개발 워크플로우

```
요구사항 분석 → 설계 → 개발 → 코드 리뷰 → 테스트 → 배포
```

#### 각 단계 상세

1. **요구사항 분석**
   - 사용자 스토리 작성
   - 수용 기준 정의
   - Jira 이슈 생성

2. **설계**
   - 시스템 아키텍처
   - 데이터베이스 설계
   - API 설계

3. **개발**
   - 코딩 컨벤션 준수
   - 단위 테스트 작성
   - 정기적 커밋

4. **코드 리뷰**
   - PR(Pull Request) 생성
   - 리뷰어 피드백
   - 수정 및 승인

5. **테스트**
   - 단위/통합/E2E 테스트
   - QA 테스트

6. **배포**
   - CI/CD 파이프라인
   - 스테이징 → 프로덕션
   - 배포 후 모니터링

### 2.3 버전 관리 (Git Flow)

#### 브랜치 구조
```
main (프로덕션)
└── develop (개발)
    ├── feature/* (기능)
    ├── bugfix/* (버그 수정)
    └── hotfix/* (긴급)
```

#### 커밋 메시지 규칙
```
<type>: <subject>

Types:
- feat: 새 기능
- fix: 버그 수정
- docs: 문서
- style: 포맷팅
- refactor: 리팩토링
- test: 테스트
- chore: 빌드/설정
```

### 2.4 CI/CD 파이프라인

#### CI (Continuous Integration)
```
코드 Push → 빌드 → 테스트 → 코드 검사 → 결과 보고
```

#### CD (Continuous Deployment)
```
Main 브랜치 변경 → 배포 전 검증 → 배포 → 모니터링
```

## 3. 비개발자 직군별 업무 프로세스

### 3.1 제품 관리 (Product Management)

#### 제품 발견 프로세스
1. **고객 조사**: 인터뷰, 시장 분석
2. **요구사항 문서화**: 니즈 정의, 범위 설정
3. **우선순위 결정**: RICE 스코어링
4. **백로그 관리**: Story 생성, 포인트 추정

#### RICE 스코어링
```
RICE = (Reach × Impact × Confidence) / Effort
```

#### 로드맵 계획
- 전략 수립 (분기별)
- 마일스톤 설정
- 기능별 스토리 정의
- 의존성 관리

---

### 3.2 디자인 (UX/UI Design)

#### 디자인 프로세스
1. **사용자 조사**: 인터뷰, 설문, 테스트
2. **와이어프레이밍**: 화면 레이아웃 설계
3. **프로토타이핑**: 목업 제작
4. **디자인 시스템**: 색상, 타이포, 컴포넌트
5. **개발 핸드오프**: 스펙 문서 작성

#### 핸드오프 체크리스트
- 컴포넌트별 치수 및 간격
- 색상값 (HEX, RGB)
- 폰트 설정
- 반응형 브레이크포인트
- 애니메이션 명세

---

### 3.3 마케팅 (Marketing)

#### 캠페인 계획 프로세스
1. **목표 설정**: KPI, 타겟 정의
2. **콘텐츠 계획**: 메시지, 콘텐츠 유형
3. **채널 선택**: SNS, 이메일, 광고
4. **일정 수립**: 공개 일정
5. **실행**: 콘텐츠 생성, 배포
6. **모니터링**: 성과 추적, 최적화

#### 주요 마케팅 지표
| 지표 | 정의 |
|------|------|
| **CAC** | 고객 획득 비용 |
| **LTV** | 고객 생애 가치 |
| **Conversion Rate** | 전환율 |
| **CTR** | 클릭율 |

#### A/B 테스트 프로세스
```
목표 정의 → 가설 수립 → 버전 준비 → 테스트 실행 → 결과 분석 → 구현
```

---

### 3.4 영업 (Sales)

#### 영업 프로세스
1. **리드 생성**: 인바운드/아웃바운드
2. **리드 검증 (BANT)**:
   - Budget (예산)
   - Authority (의사결정권)
   - Need (필요)
   - Timeline (일정)
3. **제안**: 제안서 작성
4. **협상**: 가격, 조건 조율
5. **거래 종료**: 계약, 온보딩

---

### 3.5 인사/운영 (HR/Operations)

#### 채용 프로세스
1. 채용 계획 (1주)
2. 공고 배포 (2주)
3. 서류 검토 (1-2주)
4. 면접 1차/2차 (각 1주)
5. 검증 (3-5일)
6. 제안 (3-5일)
7. 온보딩 (1주)

#### 온보딩 체크리스트
- 회사 계정 설정
- 업무 도구 접근 권한
- 업무 기기 배분
- 회사 정책 문서 배포
- 멘토 지정
- 첫 태스크 할당

#### 성과 평가 프로세스
```
평가 기간 설정 → 자기 평가 → 상사 평가 → 동료 피드백 → 면담 → 계획 수립
```

## 4. 글로벌 표준 프레임워크 및 방법론

### 4.1 프로젝트 관리 표준

#### ISO 21500
- 국제 표준화 기구(ISO) 개발
- 모든 산업에 적용 가능
- 핵심 프로젝트 관리 원칙 제공

#### PMI (PMBOK)
- 5개 프로세스 그룹, 10개 지식 영역
- PMP 자격증이 글로벌 표준
- 채용공고의 52%가 요구

#### PRINCE2
- 영국 개발, 프로세스 기반
- 프로젝트를 통제 가능한 단계로 분할
- 영국, 유럽, 호주에서 인기

### 4.2 자격증 비교

| 자격증 | 기관 | 초점 | 산업 |
|--------|------|------|------|
| **PMP** | PMI | 프로젝트 전달 | 모든 산업 |
| **ITIL** | AXELOS | IT 서비스 관리 | IT |
| **COBIT** | ISACA | IT 거버넌스 | 모든 산업 |

### 4.3 지속적 개선 프레임워크

#### Kaizen (개선)
- 일본 "변화 + 좋음"
- 점진적, 지속적 개선
- Toyota에서 대중화

#### Lean

## 5. 문화별 업무 프로세스 차이점

### 5.1 Hofstede의 6가지 문화 차원

#### 1. 권력 거리 (Power Distance)
- **높음** (인도, 중국, 한국): 계층적, 중앙화된 의사결정
- **낮음** (미국, 독일): 참여적, 평등주의적

#### 2. 불확실성 회피 (Uncertainty Avoidance)
- **높음** (일본, 그리스): 엄격한 규칙, 명확한 지침
- **낮음** (호주, 싱가포르): 유연, 위험 감수

#### 3. 개인주의 vs 집단주의
- **개인주의** (미국, 영국): 개별 성과, "나"
- **집단주의** (중국, 일본): 팀 성과, "우리"

#### 4. 장기 vs 단기 지향
- **장기** (중국, 한국): 인내, 계획
- **단기** (미국): 즉각적 결과

#### 5. 남성성 vs 여성성
- **남성적** (일본, 독일): 성과, 경쟁
- **여성적** (북유럽): 관계, 삶의 질

#### 6. 방종 vs 절제
- **방종** (호주, 미국): 여가, 즐거움
- **절제** (한국, 싱가포르): 의무, 책임

### 5.2 지역별 의사결정 스타일

#### 아시아 (일본, 중국)
- **의사결정**: 합의 기반, 느림
- **커뮤니케이션**: 간접적, 고맥락
- **리더십**: 권위적, 계층성

#### 미국
- **의사결정**: 빠름, 경영진 중심
- **커뮤니케이션**: 직접적, 저맥락
- **리더십**: 참여적, 성과 중심

#### 북유럽 (스칸디나비아)
- **의사결정**: 합의 기반
- **커뮤니케이션**: 평등주의
- **리더십**: 참여적

#### 독일
- **의사결정**: 전문가 기반
- **커뮤니케이션**: 구조적
- **리더십**: 책임감, 효율성

### 5.3 문화 간 커뮤니케이션 10가지 전략

1. **문화 지능(CQ) 개발**: 자신과 타문화 이해
2. **교육 프로그램**: 문화적 민감성 훈련
3. **커뮤니케이션 규범 수립**: 투명한 정책
4. **능동적 경청**: 완전한 이해 추구
5. **비언어적 신호 인식**: 제스처, 표정
6. **명확한 언어 사용**: 전문 용어 회피
7. **스타일 적응**: 고/저맥락 문화 고려
8. **기술 활용**: 번역 도구, 화상 회의
9. **팀 빌딩**: 문화 교류 활동
10. **포용적 환경**: 다양성 존중
